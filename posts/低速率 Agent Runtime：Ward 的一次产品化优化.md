Ward 最近又折腾了一轮。

本来按我之前的计划，下一步应该给 AI 分析加 Reviewer：Worker 先写报告，Reviewer 再检查格式、事实、指标覆盖，不合格就打回重写。这个想法很 Harness，也很符合现在大家讲 Agent 的套路。

但真准备动手的时候，我发现一个很现实的问题：大模型太慢了。

一次个股分析本来就要几十秒，如果再加一个 Reviewer，相当于每次用户点一下按钮，我让他坐在那里等两轮大模型调用。

所以这次没继续往"更聪明"的方向做，先补了一层更朴素的东西：让 Ward 在大模型慢、速率有限、调用会失败的情况下，尽量表现得像个正常产品。

---

## AI 分析异步化

Ward 之前的 AI 分析流程很直：

```
点 AI 分析 -> 后端拉数据 -> 拼 prompt -> 调大模型 -> 返回报告
```

这个流程写起来很舒服，问题是用户不会按程序员预期使用。

他可能连续点 AAPL、NVDA、TSLA，或者一个指数分析还没出来，又去点黄金分析。后端收到请求就直接开干，多个 LLM 请求同时出去。要是模型接口给力还好，一旦限速、排队、网络波动，页面上就只剩几个转圈圈。

从用户角度看，这东西就是卡了。

所以我把 AI 分析改成了 Job。这个改动看起来只是多了一层任务表，实际上把整个交互方式都改了。

现在点 AI 分析，后端不会把这次 HTTP 请求一直挂到 LLM 返回，而是先创建一个任务，把任务 ID 返回给前端。任务自己在后台排队、执行、保存结果，前端只负责订阅它的状态。

```
queued -> running -> succeeded / failed
```

这里面有几个状态：

- `queued`：任务已经创建，等后台 worker 来处理。
- `running`：任务开始执行，准备拉数据、查缓存、调模型。
- `succeeded`：报告生成完成，前端拿结果展示。
- `failed`：中途失败，错误信息也会落下来。

并发也先限制成 1，主打一个老实排队。

这个限制看起来保守，但我觉得很适合 Ward。它不是高并发服务，不需要同时给几十个人生成报告。它更像一个桌面旁边的工具：我点一个分析，它慢慢跑；我又点一个，它排到后面。这样至少不会出现三个请求一起冲出去，然后全部卡在模型接口上的情况。

任务化以后，还有一个好处：失败不会散落在各处。

以前一次请求失败了，前端大概只能拿到一个"请求失败"。现在失败会绑定在 job 上，状态是 failed，错误信息也在这个 job 里。后面要查为什么失败，就顺着 job 看，不用在浏览器控制台、后端日志、模型报错之间来回翻。

这次没有上 Redis、Celery 之类的东西，只用了 SQLite 记录 job 和事件，内存队列负责调度。听起来土，但对这个规模够用。服务重启时，没跑完的任务直接标记失败，不装作还能恢复。个人项目就别给自己加太多分布式系统戏份了。

以前是模型慢，用户跟着一起干等。现在至少能告诉他：你这个任务已经排上了，前面还有几个，当前跑到哪一步了。

## 结果缓存复用

做完队列以后，第二个问题就很明显：同样的数据不要反复调大模型。

比如我刚分析完 AAPL，过两分钟又点一次 AAPL。60 日 K 线、财务数据、新闻标题大概率都没变，再调一次 LLM 就很浪费。

这次给分析结果加了缓存。个股和指数分析缓存时间比市场总览稍长一点，模型名也放进 cache key。

模型名这个细节是后来补的。因为 Ward 已经从 MiniMax 切到了 DeepSeek，如果 cache key 里没有模型名，可能会出现一种很怪的情况：你以为自己在看 DeepSeek 生成的新报告，其实是之前 MiniMax 留下的缓存。

这种问题不一定会立刻报错，但会让调试变得很迷惑。

缓存这东西看起来普通，但在低速率环境里非常关键。有限的大模型调用，应该留给真正需要重新生成的地方。

## 进度提示与等待体验

我以前写 Ward 的时候，已经做过一些 SSE 进度提示，比如智能问答里显示工具调用状态。这次把 AI 分析也补上了类似的状态。

现在点分析以后，页面会显示：

```
任务已进入队列
正在检查缓存
正在获取行情、K线、新闻和财务数据
正在等待模型生成分析报告
分析任务完成
```

这些状态没有改变模型速度，但改变了等待体验。

用户不怕等一会儿，怕的是不知道系统到底还在不在。

这和我本地跑模型时看插座功率有点像。之前测 Qwen 本地模型，我发现功率表很有用：显卡还在吃 700W，说明模型还在跑；功率掉下来了，说明推理结束了。功率表本身不让模型更快，但它给了一个很朴素的反馈。

Ward 这次的状态提示，本质上就是给 AI 分析加了一个"功率表"。

## Agent Trace：借鉴 APM 的调试链路

做 AI 工具有个很烦的地方：出问题的时候，不知道该怪谁。

报告内容不对，可能是行情数据错了，可能是 prompt 写歪了，可能是模型自己发挥了，也可能是缓存拿错了。

这其实和以前做后端服务时遇到的问题很像。

微服务里一个请求从网关进来，可能经过用户服务、订单服务、库存服务、支付服务。接口慢了或者报错了，只看最终响应没什么用，得看整条链路：请求到了哪一层、每一层耗时多少、哪个下游报错、日志里对应的 trace id 是什么。

APM 做的就是这类事情。它不负责让业务逻辑更聪明，但能把一次请求的执行过程摊开，让你知道问题卡在哪。

Ward 这次的 Trace 基本就是借了这个思路，只是链路从微服务调用变成了 Agent 任务执行：

```text
创建 Job
-> 进入队列
-> 检查缓存
-> 获取行情 / K线 / 新闻 / 财务数据
-> 调用 LLM
-> 保存报告
```

以前 AI 分析出问题只能靠猜。现在每次分析都有一个 `job_id`，完成后报告底部会显示：

```
Job: job_xxx · 查看 Trace
```

点进去是一个单独的 Runtime 页面，可以看到这个任务完整跑了哪些步骤：

- 什么时候进队列
- 缓存有没有命中
- 数据获取用了多久
- 调 LLM 用了多久
- 传给模型的 prompt 是什么
- 模型原始返回了什么
- token 用了多少

这些信息不一定每天都看，但一旦出了问题就很有用。

比如报告里出现了一个明显不对的结论，先看 trace：如果传给模型的数据本来就是错的，那要查数据源；如果数据没问题，prompt 也没问题，那大概率就是模型生成问题；如果 trace 显示命中了缓存，那还要看是不是拿到了旧结果。

这里我特意把 Runtime 页面独立出来，没有塞在 Ward 主页面中间。主页面是给自己看行情、查股票、问问题用的，Runtime 是给调试用的。两个入口混在一起，页面会变得很奇怪。

所以现在首页底部只有一个很小的 Runtime 链接。平时不用看，出问题的时候再进去查。

这次加 Trace 以后，Ward 就没那么玄学了。至少以后看到一段离谱分析，我可以顺着 job 查回去：到底是数据不对，还是 prompt 不对，还是模型在一本正经地胡说。

## 调用统计与成本记录

既然做了 Runtime 页面，就顺手加了点统计：

- 任务总数
- 成功 / 失败数
- 缓存命中率
- LLM 调用次数
- token 数
- 平均耗时

暂时没把 DeepSeek 的调用折算成多少钱。

一方面我现在用的是套餐/订阅语境，不是纯按量付费；另一方面，各家模型价格和计费规则经常变，硬编码一个价格进去，容易让数字看起来很精确，实际没啥意义。

所以先记 token 和调用次数。等以后真有必要控制成本，再补价格表。

不过就算不算钱，统计也很有用。比如以后我可以知道：

- 哪类任务最慢
- 哪类任务最耗 token
- 缓存命中率有没有意义
- 换模型以后耗时有没有变

不然优化全靠感觉，感觉这东西和股市一样，不太可靠。

## 轻量方案与成熟框架的权衡

做完这些以后，我也想过这个问题。

LangChain、LangGraph 这类框架肯定有缓存、trace、checkpoint、持久化之类的能力。再往传统后端看，任务队列也有 Celery、Redis、RQ 这些成熟方案。

那 Ward 现在自己搞一个 Job 表、事件表、内存队列，是不是重复造轮子？

我觉得是，但问题不大。

Ward 现在要解决的问题很小：

```
点 AI 分析
不要并发把模型接口打爆
重复分析尽量复用
用户知道现在跑到哪了
我能查到原始 prompt 和返回
```

为了这点需求引入一整套大框架，项目会变得更重。Ward 本来就是个个人工具，不是公司里的统一 Agent 平台。

现在这套轻量 Runtime，刚好够用。Mini-Agent 继续负责 agent loop、tool calling、上下文压缩这些；Ward Runtime 只管产品层面的排队、缓存、trace、统计。

边界还算清楚。

以后如果 Ward 真变成复杂工作流，比如一个任务里要多轮数据收集、人工确认、重跑、版本管理，那时候再考虑 LangGraph 也不迟。

## 总结

这次优化没有让 Ward 多懂几个金融指标，也没有让它多一个 Agent 角色。

但它现在更像一个能长期用的工具了：

- 慢任务会排队。
- 重复分析会缓存。
- 等待时有状态。
- 分析完成能查 trace。
- token 和耗时有统计。

这些东西听起来都不酷，但真实产品里挺重要。

之前我更多关注 Harness：怎么约束模型、怎么设计工具、怎么让 Agent 不乱跑。  
这次更像是在补 Runtime：模型慢、接口有限、任务会失败、用户会乱点，这些现实问题要怎么兜住。

做完以后感觉也挺朴素的：Agent 产品最后拼的不只是模型能力，更多时候拼的是这些脏活累活有没有补上。
