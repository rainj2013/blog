## 人生模式：React vs Plan-And-Execute

大模型 Agent 有两种常见的推理模式：React 和 Plan-And-Execute。用人话来说，就是"见机行事"和"谋定而后动"。

我之前的工作状态，就像个 React 模式的人肉 Agent。每天接收需求、工单、生产问题，thought and action；除了需求开发、问题排查这些具体工作，也有不少精力花在信息同步和沟通对齐上。这些事当然有必要，只是受一些外部因素影响，它们有时会变得很低效：信息要多次沟通，结论要反复确认，想用一些更先进的 AI 工具和流程模式提效也不太容易，工作节奏不断被打断，人就容易陷入一种被动的忙碌和焦虑里。我不太想让自己的精力和热情长期消耗在这些地方，而是希望能在一个沟通更高效的团队里，把更多精力放到 AI 在具体业务问题中的落地上。

生活这边也说不上理想，工作日下班后窝在出租屋里，周末再回广州，生活被切成两段。在 event loop 里转久了人就麻木了，每天按部就班，自己也没有想清楚到底想要什么。

"人生规划"这词听起来很虚，但落到纸面上，其实就是两个具体又现实的问题：我想做什么样的工作，过什么样的生活。以前这些想法更多只是模糊地在脑子里转，这次我头一次花时间把它们一条条拆开，写成更具体、可执行的计划，这大概就是 Plan-And-Execute 的 plan。

## 有欲望而无行动者滋生瘟疫

> 有欲望而无行动者滋生瘟疫——《鬼泣5》V
>
> 与其感慨路难行，不如马上出发——《Dota2》克林克兹

这两句话侧重点不同，但指向都一样：动起来。

如果你的 plan 列清楚了，就赶紧 execute。现实里，不存在一个“万事俱备”的时间点，找工作面试也一样。

```
5年以上经验的开发面试会聊什么？
还要背八股文吗？
还要刷 leetcode 吗？
现在面试是不是都会问 AI 相关的问题？
```

这些问题在网上找不到标准答案，只要计划了要换工作，就直接出去面试，先看看自己在市场里的竞争力，边面试边更新调整，就像 event loop 也要执行一轮才能 observation 和 action。

## 面试观察

休息这段时间，面试了各种类型的公司，从 old school 国企到前沿 Web3 都接触过，风格差异挺大：

- 有的公司允许直接用 AI Coding Agent，也有公司严防死守，生怕你"作弊"。
- 有面试官能根据简历内容快速理解我做过的项目，找出业务价值、重难点、可能存在的瓶颈等，进而交流架构设计和功能实现方案，自然而然地也会聊到一些八股原理；又或者是构造一个场景，考察问题分析、系统设计、AI coding工具使用能力等。也有面试官只对着简历进行关键字搜索，然后开始八股文问答，像高中英语老师检查背课文一样，他提个问题，我开始吟唱，最后来一道 leetcode 草草了事。
- 几乎所有公司都会聊 AI，但大家的理解和应用大相径庭。技术社区每天能出 100 个新的模型、协议、范式，每家企业实际落地的却各不相同，面试官的认知也各有侧重。关于 AI，大家都没标准答案——就算有了，三个月后说不定也过时了。

  一线大厂聊 AI，方向会更明确一些。偏 Foundation 的岗位，主要是训练、推理和 Infra，这些都需要比较深的背景和资历。偏应用的岗位，则更多是广告投放、内容理解、效率工具 Agent 开发这些方向，这也是传统开发工程师转型比较容易上手的方向。

  AI 确实会重构很多岗位的职责、分工和工作方式。过去一个需求可能要严格拆成产品、前端、后端、测试、运维几段流程，未来这些边界会变得更模糊，人的价值也不一定再由某个固定工种来定义。也许以后不会再有传统意义上的“开发工程师”这个岗位，但互联网行业作为一种现代服务业，最终还是要服务真实业务：要有人把问题拆清楚，把工具组织起来，把方案落到业务价值上。岗位名会变，但问题不会消失，要成为能持续解决问题的人。

社招就是企业和候选人在互相寻找匹配的人和岗位，match 上了双方是能感觉到的，任一方觉得聊得不畅快，那就不是合适的选择。

## 输出型爱好

我离职后在家躺了不到三天就陷入了静态焦虑，俗称班瘾犯了。

除了准备面试，我也花了不少时间看 AI 社区的文章，想搞清现在圈子里在流行什么：Hermes、Harness Engineering、Claude Code 源码分析、Multi-Agent 等等。看完之后还是得动手，于是结合自己买美股指数宽基的需求，做了一个美股信息收集和决策辅助的 Agent——[Ward](https://github.com/rainj2013/ward-agent)，也写了几篇博客记录。

中间还给朋友做了一个小说续写 Agent，用来读取已有的人物、世界观和情节走向，再按指定风格续写后面的内容。需求不大，但能用 AI Agent 解决一个真实存在的问题，还是会让人感受到自己的价值。

围绕 AI Coding Agent，我主要在看两个问题。一个是后端项目外部依赖太多，数据库、配置、中间件这些信息不在代码里，Agent 缺少上下文就容易写得像、但不一定写得对，所以后来写了 [环境可见性：AI开发后端项目的高质量前提——以数据库为例](/?post=环境可见性-ai开发后端项目的高质量前提-以数据库为例)，也做了一个 dbctx 小工具来验证思路。另一个是 AI 一次能生成很多代码之后，质量不能只靠“再 Review 一遍代码”来兜底，更应该把需求拆成可验证的验收标准，这部分整理成了 [不要 Review AI 写的代码](/?post=不要review-ai写的代码)。

本地模型部署也重新尝试了一轮。之前我对本地小模型的印象还停留在“能跑但不好用”，这次把部署、推理速度、工具调用和实际任务效果都重新测了一遍，整理成了 [2026年了，本地小模型能给OpenClaw、Hermes当大脑了吗？](/?post=2026年了-本地小模型能给openclaw-hermes当大脑了吗)。模型参数、显存占用、响应速度、上下文长度这些东西，单独看都只是数字，真拿来做 Agent，卡在哪、慢在哪、哪里容易掉链子，感受就很具体了。

人终究是需要通过输出来摆脱虚无的，可以不上狭义的班，但不能不做广义的工作。
