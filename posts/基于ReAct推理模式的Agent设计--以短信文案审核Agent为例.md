## 1. 项目背景与用途

金融行业对外发送的短信需要严格合规审核——营销短信不能承诺保本收益、催收短信不能胁迫性表述、验证码短信必须有时效说明。

本项目实现了一个基于大模型的**短信合规审核 Agent**，输入短信内容，自动判断类型并输出结构化审核结果，包含违规点、修改建议、合规版本示例。

---

## 2. 核心能力

- **类型识别**：自动识别验证码 / 营销 / 催收 / 权益通知四种短信类型
- **多规则覆盖**：内置 5 份合规规范文档，覆盖通用红线 + 各类型专项规则
- **结构化输出**：输出 JSON 格式审核结果，便于程序解析
- **两轮推理**：Round 1 通用红线过滤 → Round 2 向量召回专项规则 → 输出结论
- **向量检索**：规则按语义召回，只加载最相关的规则片段，避免上下文膨胀

### 2.1 审核结论分级

每条短信的审核结论分为三个等级：

| 等级 | 标识 | 含义 |
|------|------|------|
| 🟢 green | 合规 | 完全符合所有规范，无任何问题 |
| 🟡 yellow | 整改 | 存在需整改项，如信息不完整、警告类问题 |
| 🔴 red | 违规 | 存在一票否决项，如伪造主体、胁迫性表述、伪造法律文书、泄露隐私等，直接判定违规 |

**一票否决项**指任何一条违规即判定 red，不受其他检查项影响：
- 伪造发送主体（冒充银行 / 监管机构）
- 胁迫性表述（威胁人身安全、恐吓上门、威胁公开隐私）
- 伪造官方文件
- 明文泄露用户隐私（身份证号、完整银行卡号等）
- 嵌入钓鱼链接

### 2.2 输出格式示例

```json
{
 "sms_type": "催收",
 "overall": "🟡 整改",
 "level": "yellow",
 "reason": "缺少客服联系方式和退订方式，'征信黑名单'表述不规范",
 "checks": [
 {
 "category": "发送主体真实性",
 "passed": true,
 "detail": "签名与实际发送主体一致",
 "suggestion": ""
 },
 {
 "category": "必含信息完整性",
 "passed": false,
 "detail": "缺少客服联系方式和退订方式",
 "suggestion": "补充客服热线：400-XXX-XXXX，退订回T"
 }
 ],
 "corrected_content": "【XX银行】您已逾期30天，欠款5000元。长期逾期将影响征信记录。客服热线：400-XXX-XXXX，退订回T"
}
```

```

```

## 4. ReAct 推理模式

### 4.1 什么是 ReAct

ReAct = **Re**asoning + **Act**ion，是一种让大模型能够"边思考边行动"的推理框架。

传统的 LLM 调用是「一问一答」：给一个 prompt，模型直接输出答案。但很多复杂问题不能一步到位——模型需要先理解问题、制定计划、执行推理、观察结果、再调整策略。ReAct 就是解决这个问题的。

ReAct 的核心是让模型在每一步输出三个部分：

- **Thought（思考）**：模型分析当前状况，决定下一步做什么
- **Action（行动）**：模型执行某个操作（如查规则、调用工具）
- **Observation（观察）**：模型根据行动结果更新认知

然后重复这个循环，直到得到最终答案。

标准格式：

```
Thought: 我需要先判断这条短信的类型...
Action: 分析短信中的关键词...
Observation: 发现"逾期""欠款""还款"等词，明确属于催收类...
Thought: 催收类短信需要检查哪些红线...
Action: 对照规范检查是否存在胁迫性表述、伪造官方文件等...
Observation: 未发现一票否决项，但缺少客服联系方式...
→ (继续直到输出最终 JSON 结果)
```

### 4.2 为什么需要 ReAct

**场景复杂时，一步到位往往不准确。**

以短信审核为例：
- 模型需要先判断类型（营销？催收？验证码？）
- 不同类型对应不同规则章节
- 规则里有优先级：一票否决项 > 禁止项 > 警告项
- 结论需要同时给出违规点 + 修改建议

如果把这些全塞进一个 prompt 让模型一步到位，容易出现：
- 类型判断和规则应用混在一起，逻辑不清晰
- 模型容易遗漏某些检查项
- 当规则较长时，上下文被稀释，重点不突出

ReAct 的好处：

| 好处 | 说明 |
|------|------|
| **过程透明** | 每一步推理过程可见，便于人工复核和调试 |
| **降低幻觉** | 模型必须"观察到"依据才能推进，不能凭空跳步 |
| **适应复杂任务** | 多步骤、需判断分支、长上下文场景下表现更稳定 |
| **便于人工干预** | 任何一步发现问题可以随时终止或修正 |
| **规则与推理分离** | 不同轮次加载不同规则，避免上下文污染 |

### 4.3 其他常见 Agent 推理模式

**CoT（Chain of Thought）**

让模型一步一步思考，不强调 Action 和 Observation，就是纯推理链。优点是轻量，适合简单任务；缺点是推理过程不结构化，模型可以"跳步"，缺乏可解释性。

**ToT（Tree of Thought）**

让模型在推理过程中探索多条分支，像一棵树一样同时考虑多个可能性，最后通过投票或评估选出最优路径。优点是适合需要探索和决策的任务；缺点是 token 消耗大，需要多次采样，成本高。

**Plan-and-Execute**

先规划再执行：模型先把任务拆解成子步骤，然后逐个执行。执行和规划是分离的。优点是复杂任务拆解后更清晰；缺点是规划阶段的输出不一定和执行阶段完全匹配，中间缺乏反馈循环。

**Reflexion（自反思）**

在推理过程中加入"反思"步骤：模型输出一个结果后，让它自我评价对不对、有什么问题，然后基于反思调整输出。优点是能自我修正错误；缺点是需要额外的反思轮次，系统复杂度上升。

### 4.4 为什么本项目选择 ReAct

| 考量 | 选择依据 |
|------|---------|
| **规则应用有顺序** | 审核必须先判断类型、再查红线、最后查专项，步骤间有明确先后依赖，不是简单的并行探索 |
| **中间结果需要可解释** | 审核过程要透明，每一步判断依据清晰可见，ToT 的多分支探索反而带来不必要的噪音 |
| **两轮设计足够** | 不需要多轮迭代修正（Reflexion），也不需要事先规划（Plan-and-Execute），Round 1 过滤红线 → Round 2 精细审核，两步足够 |
| **成本可控** | ToT 需要多次采样，成本高；ReAct 每轮一次 LLM 调用，两轮结束，token 消耗可预期 |
| **规则召回与推理解耦** | 向量检索负责找规则，LLM 负责判断，两者独立。如果未来换规则来源，推理流程不变 |

总结：**ReAct 不是最复杂的模式，但在规则应用 + 结构化输出这个场景下，它最平衡——有结构、可解释、成本可控。**

## 5. ChromaDB 向量数据库

### 5.1 为什么需要向量检索

合规规则文档较多（5 个文件，87 个 chunk），如果每次审核都把所有规则塞进 LLM prompt：
- **token 消耗大**：每次调用都传大量规则，成本高、速度慢
- **上下文稀释**：无关规则干扰 LLM 注意力，重点不突出
- **难以扩展**：规则增加后问题加剧

向量检索解决这个问题：**只召回与当前短信最相关的规则片段**，按需加载。

更重要的是，向量检索解决了**语义匹配**的问题。"上报征信黑名单"和"影响个人征信记录"字面上不相似，但语义上都是在描述同一个合规要点——普通关键词检索会漏掉，但向量检索能找到。

### 5.2 ChromaDB 简介

ChromaDB 是一个轻量级向量数据库，Python-first 设计，安装简单，无需外部服务进程。

核心概念：
- **Collection**：一组向量 + 元数据的集合，类似数据库中的表
- **Embedding**：一段文本对应的向量表示（本项目用 MiniMax embo-01，1536 维）
- **Query**：用向量找最相似的 K 条记录

### 5.3 稠密向量与稀疏向量

向量检索中有两种主流向量类型：

**Sparse Vector（稀疏向量）**：向量中大部分维度为 0，只有少数维度有值。典型代表是 TF-IDF、BM25 算法生成的向量。适合精确关键词匹配——查询中出现了某个词，对应的维度就是非零的，能直接命中。但无法捕捉语义相似性，"上报征信黑名单"和"影响个人征信记录"字面不同，稀疏向量表示也不同，检索就会漏掉。

**Dense Vector（稠密向量）**：整个向量几乎每个维度都有浮点值，所有维度共同参与相似度计算。模型会把文本的语义压缩到连续向量空间中，相似的语义会落在相近的位置。即使关键词不同，只要意思相近，向量空间中的距离也会很近。

举例对比：

| 文本 | Sparse 表示 | Dense 表示 |
|------|-----------|-----------|
| "上报征信黑名单" | [0, 0, 1, 0, 0, 0, ...] | [0.12, -0.34, 0.78, -0.21, ...] |
| "影响个人征信记录" | [0, 1, 0, 0, 0, 0, ...] | [0.11, -0.31, 0.82, -0.19, ...] |

Sparse 下两个向量完全不同；Dense 下两个向量非常接近，因为语义上它们在说同一件事。

**各自适用场景**

Sparse 向量适合：**精确关键词检索**。当查询明确包含某个词，需要找到包含该词的文档时，Sparse 表现更好。例如查"征信黑名单"这个具体表述，TF-IDF 能直接命中包含这个词的文档。而且 Sparse 可解释性强——哪个维度对应哪个词一目了然。

Dense 向量适合：**语义相似性搜索、多语言或跨语言检索、查询与文档表述不一致的场景**。当关键词不完全匹配但意思相近时，Dense 能捕捉语义关联。本项目属于后者：规则中写的是"严重影响个人征信记录"，实际短信写的是"上报征信黑名单"，字面不同但语义相同，需要 Dense 来召回。

**本项目选择 Dense 向量（MiniMax embo-01）**，因为短信合规审核依赖语义理解：违规表述方式多样，"追究法律责任"和"可能面临法律诉讼"意思相近但字面差异大；规则中的禁止条款和实际短信的表述方式也不完全一致，需要语义层面的匹配能力。

### 5.4 索引构建

规则文档按 `##` 标题切分成 chunk，每个 chunk 调用 embedding API 转成向量，然后存入 ChromaDB，附带元数据（来源文件、分类、章节名）。

持久化到本地目录 `.chroma_db/`，无需手动管理。

### 5.5 检索流程

待审核短信先转成向量，在 ChromaDB 中用余弦相似度搜索，召回 Top-K 最相关的规则片段。同时对同类型规则（如"催收"）加权优先。

```python
results = collection.query(
 query_embeddings=[query_emb],
 n_results=k * 3,
)
# 距离转相似度 + 同类型加权 → 排序取 Top-K
```

### 5.6 向量数据库的作用

向量数据库的核心价值在于**语义检索**——通过向量相似度而非关键词匹配来找内容。一段文本转入量后，相似的意思会落在向量空间里相近的位置。检索时把查询文本也转成向量，在向量空间中找到最近的 K 个，数据就回来了。

ChromaDB 负责：向量存储、持久化、最近邻搜索（HNSW 索引）、元数据过滤。Embedding 负责：文本→向量转换（本项目用 MiniMax embo-01）。
