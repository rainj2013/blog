最近看了 MiniMax 那篇《[一个 AI 还是不够](https://mp.weixin.qq.com/s/TIL7o92f71DsPPLWT4_37A)》，感觉很有价值，在 Ward 上做了一轮实践。我一直有个疑问：Multi Agent 是必须的吗？到底怎么落地才有用？那篇文章给了我一些具体的思路。

---

## 单 Agent 的问题

Ward 原来就是个典型单 Agent 投研助手。用户问一句，它判断要不要调工具，调完组织回答。简单问题上很好用：

- 纳指最近怎么样
- AAPL 今天涨了吗
- 黄金盘后有没有变化

路径很短，一个 Agent 调几个工具就能答。

但用户一旦问：

> 帮我从各个角度对比 NVDA 和 AAPL 两只股票

事情就复杂一点了，算是个小型投研任务。要分别拿两只股票的数据，看行情、看趋势、看估值、看数据缺口，最后再横向比较。如果还是一个 Agent 从头跑到尾，它就要同时干一堆活：

- 自己拆任务
- 自己查数据
- 自己比较
- 自己写报告
- 自己判断报告靠不靠谱

这就是 MiniMax 说的：一个 Agent 同时当选手和裁判。

从用户视角看也很明显——他根本不知道系统在干嘛。是在查 NVDA 还是在查 AAPL？是数据源慢还是模型在生成？报告有问题，是数据错了还是总结错了？

Ward 跑在微信上，这个问题还多了一层——用户问完一句话，Agent 半天不回。是在查数据还是卡住了？用户只能干等。文章管这叫"秒回和长任务之间的矛盾"：要给个好答案需要时间，但让用户等着感受就很差。

以前我的思路是继续补 Prompt 和 Tools，让 Agent 自己做 Plan，逐步调用合适的 Tool 去实现。看完 MiniMax 那篇文章以后，我觉得方向得换一下。给复杂任务一个更稳定的执行结构，别让一个 Agent 继续"努力"了。

---

## 我理解的 Agent Team

MiniMax 文章里有一句我很认同：

> 多 Agent 系统不是 Prompt/Skill 编排，而是一套需要持续运行和维护的基础设施。

很多所谓多 Agent 方案，其实就是给模型写几个角色 Prompt。一个叫 Planner，一个叫 Writer，一个叫 Reviewer。看上去像团队，跑起来还是一个语言模型在不同角色之间切来切去。

角色名字不值钱，背后的机制才难：

- 谁判断任务值不值得拆？
- 谁决定拆成几步？
- 每个 Worker 看哪些上下文？
- 中间结果怎么传？
- 谁来验收？
- 失败了怎么办？
- 整个过程能不能回放？

这些不解决，多 Agent 就是个更复杂的黑盒。所以 Ward 这次没做通用 Multi Agent 平台。那太重了，也没必要。我只是把"多股票对比"这一个场景，用 Multi Agent 的思路做了轻量实现，叫它 Team 化可能更合适。

---

## 不是什么都要上 Team

第一个原则：不要为了 Team 而 Team。

用户问"纳指最近怎么样"，单 Agent 搞定。调工具、读数据、总结回答，路径很短，没必要搞后台任务。但用户问"对比 NVDA 和 AAPL"，系统就该走 Team Job 路径。

分流逻辑不靠大模型，用了一个规则引擎：

1. 检查消息里有没有"比较"、"对比"、"compare"、"vs"这类关键词
2. 把消息里全大写的字母提出来，当作候选股票代码
3. 逐个查行情，能查到有效行情才算数
4. 有效代码在 2-6 个之间，才认定为多股对比

命中就走后台 Team 流程，没命中就交给常规单 Agent。路由本身不需要大模型参与，响应快，可预测。用户问"对比 NVDA 和 AAPL"不会被误解，模型抽风也不会把简单问题路由到 Team。

多 Agent 应该按需启用，不该是默认行为。任务短、风险低、路径明确，就别搞复杂。任务长、数据多、需要交付质量，才值得拆。

---

## Leader：把任务变成流程

Ward 的多股对比里，Leader 不直接干活。它先负责把用户的一句话变成一个后台任务：

```text
用户：帮我从各个角度对比 NVDA 和 AAPL
Leader：这是一个多股票对比任务，需要分别收集数据，再汇总分析
```

Leader 更像控制面。路由已经帮它确定好要对比哪些股票了，它不处理原始数据，而是给每只股票分配一个 Worker 去独立收集数据，等所有 Worker 完成后，再拿它们产出的结构化摘要做横向对比。

和单 Agent 最大的区别是：系统不再把用户的一句话直接塞给模型硬跑，而是先转成一个可追踪的流程。对用户来说，看到的是"任务已创建"。对系统来说，拿到的是一个 job，有状态、有事件、有 trace。看起来不复杂，但产品形态变了。以前是请求，现在是后台执行过程。

---

## Worker：只做自己的那部分

每只股票对应一个 Worker。Worker 不写最终报告，也不判断哪只股票更好。只管自己这只股票的数据整理清楚。比如：当前价格、今日涨跌幅、近期走势、均线状态、K 线数量、数据是否完整、哪些字段缺失。整理成结构化摘要，交给 Leader。

这里关键的是上下文隔离，并行反而是附带的。单 Agent 做对比时，所有股票的数据、工具结果、分析过程全混在一个上下文里。数据一多，模型容易分心，你也搞不清某个结论到底从哪来的。

现在 Worker 只看自己负责的股票，不需要知道别的，也不需要做最终判断。这很像传统工程里的模块边界——每个模块先把自己的事做好，再交一个清晰的中间结果上去。Agent 系统也一样。上下文不是越多越好，很多时候恰恰要少一点、窄一点、明确一点。

---

## Leader 汇总：只看摘要，不吃原始数据

Worker 完成后，Leader 做横向比较。这里我刻意没让 Leader 直接读原始数据，只读 Worker 的结构化摘要。

这个设计来自 MiniMax 提到的一个成本问题：多 Agent 不是免费午餐。任务拆出去以后，至少多了三类开销：

- **交接成本**——Worker 产出的原始数据不能直接丢给下一个 Agent，要转成摘要对方才能用。每一次交接都在翻译一次。
- **共享成本**——直觉上让所有 Agent 看到所有信息最安全，但每多共享一段内容，每个 Agent 每轮都多付一次 token。按需加载比全量共享划算得多。
- **聚合成本**——派几个 Worker 并行查数据不难，难的是把几份结果合在一篇报告里，事实一致、风格统一。这一步没有捷径。

为了"保险"让每个 Agent 看到所有信息，系统很快会变重。Ward 选了个更克制的方式：

```text
Worker 读原始数据
Worker 产出摘要
Leader 只读摘要
Leader 生成对比报告
```

不一定是最强的方案，但比较稳。报告出问题，可以回到 trace 看：是某个 Worker 的摘要有问题，还是 Leader 比较有问题。以前单 Agent 一口气写完，出了问题只能猜。

---

## Verifier：别让模型自己说自己合格

MiniMax 反复提到 Verifier。我觉得这是 Agent 从 demo 走向产品的关键一步。

模型写出一篇报告，不等于报告能交付。以前 Ward 主要靠 Prompt 要求模型：

> 不要编造数据。

有用，但不够。

现在 Ward 加了一个确定性的 Verifier。它不试图理解金融市场，也不调另一个大模型来审稿。先做一些很朴素的检查：

- 报告是不是空的
- 是不是明显太短
- 有没有模板占位符
- 关键章节有没有缺失
- 多股对比有没有覆盖所有标的

看起来很简单，但它把"模型生成了"推进到了"系统检查过了"。文章里说的 Verifier 是 Worker 和 Verifier 之间多轮对抗——Worker 写报告，Verifier 打回去让改，改完再审。Ward 暂时没做到那一步，用了更轻量的方式：一次性确定性检查。够用，但确实还有迭代空间。

我现在越来越觉得，Agent 产品不能只依赖模型的自我判断。模型很擅长把话说圆，但系统需要有外部反馈、有验收条件、有失败状态。Verifier 的意义就是把"完成"变成"可交付"。

## Runtime：复杂系统必须可观察

这次改造里，我最满意的是 Runtime，多股对比本身反而在其次。一个 Agent Team 如果不可观察，就只是一个更复杂的黑盒。

Ward 的 Team Job 会记录完整 trace：

- 什么时候创建任务
- Leader 什么时候生成计划
- 每个 Worker 是否完成
- 每个 Worker 拿到了多少数据
- Leader 什么时候开始汇总
- Verifier 是否通过
- 失败时错误是什么

Runtime 页面还加了一个 Team Overview：

```text
Leader
Workers
Verifier
```

用户不一定每天看这些东西，但一旦任务慢了、失败了、结果奇怪了，就很重要。

我一直觉得做 Agent 产品要多借鉴传统后端工程。MiniMax 文章也提到，他们受了 Harness 思想的启发——Agent 不光要干活，底层还得有状态机来管理什么时候该执行、什么时候该验证、什么时候该重试，而不是靠模型自己判断"下一步该干嘛"。以前做服务，请求慢了看 trace，接口错了看日志，任务失败了看状态机。Agent 也一样。模型越强，越应该把过程摊开。否则用户只会看到一个更会说话的黑盒。

## 一些体感

做完以后，对 Agent Team 的理解更实际了一点。它只是为了让系统在复杂任务里更像一个正常的软件产品，不是为了看起来更"智能"。

单 Agent 最大的问题，是它承担了太多不该混在一起的职责。让一个 Agent 同时计划、执行、总结、验收、保存状态，本质上是在把很多工程问题都丢给模型。短任务还行，长任务就开始露馅。

Team 化说到底就是：把职责拆开，把状态写下来，把过程摊开，把验收条件放到系统外面。比"多搞几个模型"要复杂得多。这套东西听起来不像 AI，更像工程。但这恰恰是 Agent 产品真正要补的部分。

---

## 总结

MiniMax 那篇文章把两件事讲清楚了：什么时候该拆、拆了之后怎么跑。

什么时候该拆？单 Agent 在任务变长、信息变多、需要交付质量的时候，会同时暴露出"不知道自己什么时候该停、干到一半容易跑偏、过程不可追溯"这几个问题——Ward 的多股对比全都踩中了。

拆了之后怎么跑？Leader 负责计划和汇总，Worker 各自收数据互不干扰，Verifier 从外部做检查，不靠模型自己说"做完了"。Ward 就是顺着这个框架做的轻量实现。

简单问题交给单 Agent，复杂对比走 Team Job：

- Leader 负责计划和汇总
- Worker 负责独立收集数据
- Verifier 负责质量检查
- Runtime 负责过程可观测

一个 Agent 当然还能做很多事。但任务越复杂、链路越长、交付质量需要把关时，一个 Agent 确实不太够了。

多 Agent 不是目的。只是想让复杂任务在 Ward 这里没那么黑盒、没那么容易卡住、没那么依赖模型的自我感觉。
