前段时间我写过一篇 [《不要 Review AI 写的代码》](/posts/不要%20Review%20AI写的代码/)，核心观点是：AI Coding 时代，质量验证的主线不应该继续放在逐行看 diff 上，而应该放在验收标准和验证证据上。文章发出去后，很多网友持不同意见。大家的核心顾虑也很明确：严肃生产代码还是应该经过人工评审，否则质量怎么保证？这个担心我也理解，我之前反对的是把质量保障主要放在逐行看 diff 上，不是说人可以完全不管代码质量。人工评审仍然有价值，尤其是在架构边界、安全风险、复杂业务规则和高影响变更上。

所以这篇文章不继续争论“到底需不需要人工评审”。我更想讨论的是另一个问题：如果我们承认 AI 生成业务代码的速度明显快于人工，单纯逐行 Review 会越来越吃力，那之前那套“验收标准 + 验证证据”的方案还能怎么继续优化，能不能进一步变成一套更系统的代码质量验证体系。当然，这套方案本身是否可靠，也不能只靠我写文章论证。更好的方式是找一个能承受风险的真实项目，连续几个版本采用这种流程，再和传统评审流程对比线上 bug 率、返工率、交付周期、回归问题数量等指标。只有用量化结果评估过，才能判断它到底是在提升质量，还是只是换了一种更复杂的流程。

最近看到一篇[《AI Agent & Skill 测评方案及落地实践》](https://mp.weixin.qq.com/s/PUbGqheJhFMmb6hGj1ZtOw)，里面有几个点对我很有启发：

- 测评不能只看最终结果，要有评分器组合
- 不能只做一次验证，要有用例集、基线和回归
- 不能只判断“这次能不能用”，还要持续比较不同版本的表现

不过那篇文章评估的是 Agent 和 Skill，本文讨论的是 AI 生成出来的业务代码质量，也就是实际项目中的代码改动是否可靠。

## 这和 AI 的关系在哪里

这里需要先说清楚一点：后面提到的用例集、回归、基线、质量门禁，并不是 AI 出现后才有的新东西。成熟的软件工程本来就应该做这些事。如果把这篇文章理解成“AI 时代发明了一套全新的代码质量方法”，那就偏了。

AI 带来的变化主要不在方法本身，而在工程节奏和风险分布上。过去人写代码慢，很多质量问题会在设计、编码、调试、Review 的过程中被逐步暴露；现在 AI 可以很快给出一版实现，甚至连测试也一起补上，代码生产速度突然快了很多，但业务理解、边界确认、回归验证并没有同步变快。于是瓶颈从“怎么把代码写出来”转移到了“怎么证明这些代码真的可靠”。

更麻烦的是，AI 写出来的代码和测试往往看起来都很完整。如果需求理解错了，它可能同时写出错误实现和能通过的错误测试。这个时候，传统那种“看起来有测试、CI 也是绿的”就不够了。人的注意力更应该放在验收标准、关键边界、回归证据和高风险判断上。这也是为什么我会借用 Eval 的说法：不是因为这些质量手段只属于 AI，而是因为 AI Coding 把它们的重要性和工程化程度都往前推了一步。

## 上次方案解决了什么

上次文章的主要问题意识是：不要让人被 AI 生成的大量代码淹没。代码看起来完整，命名也像那么回事，甚至测试也能通过，但这并不等于业务需求真的被满足。真正需要确认的不是“这段代码读起来是否合理”，而是“需求有没有被可靠证明”。

所以我当时提出，人的注意力应该往上游移动：

- 不再主要 Review 代码实现
- 先 Review 验收标准
- 再 Review 关键测试场景
- 最后看每条需求有没有证据覆盖

这套流程的价值在于，它把问题从“代码看起来对不对”转成“需求有没有被证明”。比如一个 VIP 折扣需求，不是让人先去看 `if amount > 100` 写在哪，而是让验收矩阵明确记录：

- `amount = 100` 时不打折
- `amount = 101` 时打折
- 普通用户不打折
- 已使用优惠券时不能叠加

然后每条验收标准都要对应测试、命令、日志或人工验收证据。这比“AI Review 通过”要可靠很多，因为它至少把业务正确性从代码阅读里抽了出来，变成可检查的证据。

## 上次方案的不足

但回头看，上次方案还是不够完整。它更像一个 Requirement Verification Harness：围绕单次需求，拆验收标准、生成测试、收集证据、输出报告。这个方案能解决“一次改动是否满足需求”的问题，但还没有形成一套面向业务代码质量的长期验证体系。参考文章对我的启发，主要也是从用例集、验证方式、基线和工程化执行这几个角度，把一次性验证推进成持续测评。

第一个不足是缺少回归沉淀。如果今天改了折扣逻辑、明天改了订单状态机、后天重构了支付回调，我们需要回答的不只是“这次需求有没有通过”，还包括：老场景有没有退化？历史 bug 有没有复现？关键边界是否仍然成立？上次方案更关注单个需求的证据，没有很好地把这些证据沉淀成后续可重复运行的回归用例和行为基线。

第二个不足是验证维度偏窄。业务代码质量不只是“功能测通了”。还要看边界条件、异常路径、数据兼容性、权限约束、并发风险、性能影响、可维护性和迁移风险。比如一个接口改动，主路径测试通过了，但老客户端传旧字段会不会坏？失败重试会不会重复扣款？数据库迁移能不能回滚？这些都属于业务代码质量的一部分。

第三个不足是没有把验证证据分层。上次方案的重点是用测试和验证命令收集证据，这个方向没有问题，但还可以继续细化：哪些证据可以作为硬门禁，哪些只是辅助参考，哪些关键规则必须人工确认。比如构建、单元测试、接口契约、数据库迁移检查可以自动化；验收标准是否表达了真实业务、关键边界有没有漏掉、高风险场景是否需要补人工验收，则应该明确标出来。否则很容易只看到“测试通过”，却不知道这些测试到底覆盖了多少业务风险。

第四个不足是缺少工程化的持续比较和门禁。一次需求做得再认真，如果验证只停留在本次报告里，没有在后续版本继续运行，没有形成趋势对比和准入规则，它的价值就会很快消失。业务代码质量不是某次上线前的一次性验收，而是每次改动都要面对同一套历史约束。这里才是我觉得可以借鉴 Eval 思想的地方：不是把业务代码当成 Agent 来测，而是把代码质量验证做成可重复、可比较、可持续演进的工程系统。

## 把代码质量验证变成可重复过程

借用 Eval 的概念，重点不是给代码打一个漂亮分数，而是把一次业务代码改动放进可重复的验证过程里。它可以理解成：

> Code Eval = 给定需求和代码改动，运行一组可重复的检查规则，收集验证证据，判断这次改动是否满足业务质量要求。

放到代码场景里，一次 Code Eval 至少应该包含几类材料：需求描述、验收标准、代码 diff、项目上下文和历史质量用例。验证动作也应该来自项目自己的工程体系，比如单元测试、集成测试、契约测试、端到端测试、数据库迁移检查、静态扫描、性能基准、关键路径人工验收。最终结论不应该是“模型觉得可以”，而应该是每条业务规则、每个风险点、每个质量门禁分别对应了什么证据。

从这个角度看，Eval 这个词的价值不在于它多新，而在于它提醒我们：代码质量验证不能只停留在“这次测试过了”，还要有用例集、基线、分层验证方式、报告和回归。它要回答的问题也很具体：这次 AI 生成的代码满足了哪些需求？哪些需求没有证据？哪些历史场景被重新验证？哪些风险只能靠人工判断？相比上一个版本，质量有没有退化？

## 三类验证方式：把判断交给合适的对象

参考 Agent 测评里的评分器组合思路，业务代码质量验证也可以拆成三类验证方式。但这里不是为了评价 AI，而是为了判断代码改动是否可靠。

第一类是确定性检查，也就是所有能用程序稳定判断的验证。这部分应该是主力，而且应该尽量做成硬门禁：

- 编译和构建是否通过
- 单元测试、集成测试、端到端测试是否通过
- API 输出是否符合 Schema
- 数据库迁移是否可执行、可回滚
- 关键配置是否缺失
- lint、类型检查、静态扫描是否通过
- 覆盖率是否低于阈值
- 性能基准是否明显退化
- 安全扫描是否发现高危问题

第二类是 Rubric 检查，也就是用结构化标准判断那些很难完全写成脚本的质量问题。比如一次 AI 生成的重构，测试都通过了，但是否贴合项目原有分层？是否引入了不必要的抽象？是否把业务规则散落到多个地方？是否为了测试通过写了脆弱实现？这些问题可以用清单、人工抽查，也可以让 LLM-as-Judge 辅助判断，但前提是 Rubric 要具体，输出要结构化，不能只让模型说“整体不错”。更合理的输出应该是：是否符合架构约定、是否存在过度设计、是否遗漏异常处理、是否引入可维护性风险，以及每一项的理由和证据位置。

第三类是人工确认。人工评审不应该完全退出，但它的位置要更清楚：人不需要把所有 AI 生成代码都逐行看完，而应该重点看高风险判断。比如验收标准是否正确，关键测试场景是否覆盖，复杂业务边界是否被误解，架构边界是否被破坏，安全和资金相关逻辑是否有风险。如果用了 LLM 辅助 Rubric 检查，人工还需要抽样校准它是否靠谱，发现误判后调整规则，而不是每次都重新从头做全量判断。

## 业务代码质量到底要评什么

如果把评估对象拉回业务代码，维度就会比“代码能不能跑”更丰富。第一是功能正确性，也就是需求有没有被满足。这个维度最适合用验收标准表达：主路径是什么，边界值是什么，异常输入如何处理，旧行为是否保持兼容。AI 写出来的测试不一定可靠，因为它可能和错误实现互相证明，所以验收标准必须先被人确认，再由测试和验证命令去覆盖。

第二是回归安全。业务代码最怕“新需求做完了，老功能坏了”。所以每次 AI 生成代码后，不应该只跑新测试，还要跑和改动相关的历史用例、线上 Bad Case、核心链路回归。如果一次线上问题被修复，它就不应该只停留在事故复盘里，而应该进入回归用例集。这样下次 AI 再改相关代码时，历史问题会自动站出来拦它。

第三是边界和异常。很多业务 bug 不发生在 happy path，而发生在空数据、重复请求、并发提交、权限不足、外部服务超时、历史数据格式不一致这些地方。AI 很容易把主路径写得很顺，但遗漏这些不显眼的失败路径。代码质量验证体系要专门覆盖这些场景，而不是只看“正常情况下能不能跑通”。

第四是可维护性。可维护性不容易量化，但不能完全不管。AI 生成代码有时会为了局部需求复制逻辑、绕过已有抽象、制造隐式耦合，短期测试能过，长期会变成债务。这部分可以结合静态规则、架构约束和 Rubric 检查来判断，比如是否复用了已有领域服务，是否把业务规则集中表达，是否引入重复实现，是否破坏模块边界。

第五是变更风险。不是所有代码改动风险都一样。改 UI 文案、改报表字段、改支付链路、改权限判断，应该进入不同级别的验证流程。高风险改动需要更强的门禁，可能包括更多回归用例、更严格人工确认、更完整的灰度和监控。低风险改动则不应该被同样重的流程拖慢。

## 用例集：把业务经验变成质量资产

上次方案更像是针对一次需求临时拆验收标准，这次应该更进一步，把验收标准沉淀成用例集。一个好的业务代码质量用例集，至少应该包含四类。第一类是需求主路径用例，验证新功能最核心的业务结果。第二类是边界用例，覆盖金额阈值、状态流转、空值、最大值、历史数据、权限边界。第三类是异常用例，覆盖外部服务失败、重复请求、并发冲突、数据库约束、非法输入。第四类是历史 Bad Case，用线上真实发生过的问题防止回归。

这个用例集不一定都表现为单元测试。它可以是单元测试、集成测试、API 测试、Playwright 脚本、数据库断言、日志断言，也可以是人工验收清单。关键不在于形式统一，而在于每个用例都能回答一个业务质量问题：它验证的是哪条规则？覆盖的是哪个风险？失败后说明什么？和哪次需求或哪次线上事故有关？

用例集还要分层。P0 用例是核心业务和历史事故，应该每次都跑；P1 用例是高频路径和重要边界，可以在合入前或夜间跑；P2 用例是低频组合和探索性场景，可以定期跑或在大改动时跑。否则用例越积越多，最后每次验证成本太高，团队又会回到“选择性跑一下”的状态。

## 基线：让代码质量有参照物

基线不是 Agent 测评专属概念，业务代码质量同样需要基线。对代码改动来说，基线可以是一组历史已确认正确的行为：某个接口在这些输入下应该返回什么，某个状态机应该如何流转，某个数据库迁移前后数据应该如何保持一致，某个核心链路的性能大概在什么范围内。没有基线，我们只能判断“这次有没有报错”；有了基线，才能判断“这次有没有偏离过去已经确认正确的行为”。

基线可以来自测试快照、契约文件、Golden Case、性能指标、线上真实样本，也可以来自人工确认过的验收记录。它不是永远不变，业务变了，基线也要更新。但更新必须是显式动作，要说明为什么旧行为不再适用，而不是因为 AI 改完代码后测试结果变了，就顺手把新结果当成正确结果。

这里特别适合用在 AI 生成代码场景。AI 很擅长改实现，但它未必知道哪些旧行为不能破坏。基线的作用就是把这些“不能破坏的业务事实”固定下来，让每次代码生成都必须重新面对它们。

## 过程证据：不要只相信最终测试通过

虽然本文不评估 Agent 本身，但在 AI 生成代码的场景里，过程证据仍然有价值。这里的过程证据不是为了评价“AI 聪不聪明”，而是为了判断这次代码改动是否可信。比如它有没有运行项目声明的测试命令，失败后改了什么，是否只跑了单个 happy path，是否修改了无关文件，是否更新了迁移脚本，是否补了对应文档或契约，是否有证据覆盖每条验收标准。

最终测试通过当然重要，但它不是全部。一个 AI 生成的改动如果只补了和实现完全同构的测试，或者只跑了局部测试，或者绕过了已有领域服务，即使当前 CI 是绿的，也应该被标记为风险。过程证据的价值就在这里：它让报告不只是写“通过”，还写清楚“靠什么通过”“哪些地方没验证”“哪些判断需要人工确认”。

所以，这里的 Trace 可以弱化成代码验证流水线里的 evidence log：需求、验收标准、测试计划、执行命令、命令结果、覆盖关系、未覆盖风险、人工确认点。它不需要复刻 Agent 测评里的完整执行轨迹，但要足够支持质量判断和问题追溯。

## 优化后的流程

如果把上次的方案改成更面向业务代码质量的流程，我会这样设计：

1. 根据需求拆出验收标准，人工确认关键业务规则和边界。
2. 根据验收标准生成验证计划，明确哪些用单元测试、哪些用集成测试、哪些需要人工验收。
3. 标记本次改动风险等级，比如普通功能、核心链路、权限、安全、资金、数据迁移。
4. AI 生成或修改业务代码，同时补充对应测试和验证证据。
5. 运行确定性检查，包括构建、测试、Schema、迁移、静态扫描、性能基准等。
6. 对可维护性、架构贴合度、异常处理完整性等问题运行 Rubric 检查。
7. 生成验收矩阵，记录每条验收标准对应哪些证据，哪些仍然缺失。
8. 将关键用例纳入回归集，和历史基线对比，检查是否破坏旧行为。
9. 人工只重点复核高风险规则、缺失证据、Rubric 低分项和核心业务边界。
10. 上线后把线上 Bad Case 继续沉淀进用例集。

这套流程的重点不是把 Code Review 取消掉，而是重新分配人的注意力。人更应该判断“什么算对”“哪些风险不能漏”“哪些证据足够可信”，而不是把主要精力消耗在 AI 生成的大量样板代码上。代码当然还可以看，但代码阅读应该服务于风险判断，而不是承担全部质量保障责任。

## 这套方案仍然有什么不足

即使把验证体系从单次需求升级到用例集、基线和分层验证方式，也仍然有很多问题。第一个问题是用例集会过拟合。只要有固定用例，AI 和开发流程都可能越来越擅长通过这些用例，但真实业务场景仍然会冒出新问题。所以用例集必须持续吸收线上 Bad Case，也要定期补充挑战用例和隐藏用例。

第二个问题是 Rubric 检查不稳定。让模型评价可维护性、架构贴合度、异常处理完整性，有价值，但不能无脑相信。它适合作为风险提示和辅助审查，不适合作为唯一硬门禁。真正高风险的问题，仍然需要人工抽查和校准。第三个问题是维护成本。基线、用例、验收矩阵、验证规则都需要维护，如果流程太重，团队会绕开它。所以落地时一定要分层，先覆盖核心链路和高风险场景，不要一开始就追求全量完美。

第四个问题是它不能替代线上质量体系。再完整的离线验证，也覆盖不了所有真实用户输入、真实数据状态和真实外部依赖。灰度、监控、告警、回滚、事故复盘仍然必须存在。代码质量验证体系只能减少问题进入生产的概率，不能保证永远没有问题。

## 回到“不要 Review AI 写的代码”

现在再看“不要 Review AI 写的代码”这句话，我觉得可以说得更准确一点：不是不看代码，也不是把业务代码质量交给 AI 自己判断，而是不要把逐行阅读 AI 生成代码当成质量保障主线。真正的主线应该是：

- Review 需求和验收标准
- Review 高风险测试场景
- 用确定性检查做硬验证
- 用业务用例集做回归
- 用基线发现行为退化
- 用 Rubric 检查辅助判断可维护性和架构风险
- 用人工复核高风险规则和缺失证据

这套体系比单纯 Code Review 更重，但它解决的是另一个层级的问题：不是“这段 AI 写的代码看起来怎么样”，而是“实际项目中的这次代码改动有没有足够证据证明它是正确、可靠、可维护的”。AI Coding 的效率提升已经发生了。接下来真正拉开差距的，可能不是谁更会让 AI 写代码，而是谁先把业务代码质量验证做成一套可持续运行的工程系统。因为当代码生成越来越便宜，验证业务代码是否真的可靠就会越来越值钱。
