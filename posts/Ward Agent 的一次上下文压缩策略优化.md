最近看了腾讯技术工程的一篇文章：[《横向拆解Claude Code、Codex等六大Agent上下文压缩策略后，我们做了第 7 个》](https://mp.weixin.qq.com/s/BQwyvE2qIfguwKk63F3LZw)，里面有不少关于 Agent 上下文压缩的实践理念，对我启发很大。读完以后，我也借着这篇文章提供的框架，给 Ward 做了一轮上下文管理升级。这篇文章横向比较了六种主流 Agent 的上下文压缩方案，并在这个基础上提出了第七种做法。它讲得很具体：哪些内容该保护，哪些内容可以压缩，为什么不要等到上下文快爆了才一次性全量摘要，以及不稳定的上下文改写为什么会影响 prompt cache。

Ward 原本有套简单的上下文管理机制，能解决最基础的连续对话和环境信息复用问题。但再往下做，就会遇到一些更细的问题：哪些内容应该持久化，哪些内容应该压缩，哪些内容必须保留原文，以及这些动作怎么在产品里展示出来。

很多 Agent 产品都会遇到类似问题：有一部分上下文来自用户对话，有一部分来自外部环境或工作台状态，还有一部分来自工具调用结果。短期看，把这些内容都塞进 prompt 也能跑；对话一长，问题就开始出现。

Ward 原来已经做了三件事：

- 环境信息会带进智能问答，减少重复工具调用。
- 每次对话会加载最近一段聊天历史，保证基础连续性。
- Mini-Agent 在 token 超限时，会把运行过程压缩成摘要，避免一次长任务把窗口撑爆。

这些机制有一些用。用户问一个和当前环境相关的问题，Agent 可以优先复用已经加载的环境信息；长一点的 agent loop 也不会轻易因为 token 过多直接断掉。那篇文章把几种主流 Agent 的取舍摊开以后，这轮升级的方向也更容易收敛。原来的机制更偏“局部处理”，这次往会话级维护推进。

这次升级是在原来已有机制上继续往前推进：

- 临时摘要，推进到持久摘要。
- 原始环境信息，推进到环境摘要。
- 后端内部动作，推进到前端可观测状态。
- 动态 system prompt，推进到稳定 prompt 前缀。

---

## 原来的机制哪里不够

很多带工作台的 Agent 都有类似设计：已经加载的环境信息，会带进聊天上下文。比如用户正在看某个任务、文档、报表或分析结果，然后在旁边问：

> 刚才这个结果怎么看？

理想情况下，Agent 不需要重新调工具，直接读已有环境信息回答就行。这个设计本身是对的，也确实减少了很多重复调用。问题出在粒度和生命周期。环境信息里往往不只有一条数据，还会有原始明细、工具结果、报告内容、用户选择的对象、历史对话。一次普通聊天，背后可能带着一堆上下文。

原来的处理方式大概是：

- 环境信息格式化后拼进 system prompt。
- 历史消息只取最近 20 条。
- token 超限时，在当前 Mini-Agent 实例里做一次运行过程摘要。

这里有几个具体问题。第一，环境信息会让 system prompt 每轮变化。环境状态、工具结果、报告内容只要变一点，system prompt 就跟着变。这个结构对 prompt cache 不友好。第二，最近 20 条之前的历史只是自然消失。它没有被整理成“对继续工作有用的状态”，直接不进入下一轮 prompt。如果用户早些时候说过自己的偏好或约束，后面就可能丢掉。第三，Mini-Agent 的摘要更偏运行时保护。它能解决一次 agent loop 内 token 快超的问题，但摘要没有作为会话状态持久保存。下一次请求重新从存储里加载历史时，这个摘要并不会自然接上。第四，整个过程对用户不可见。上下文到底加载了多少、环境信息压没压、有没有触发摘要，全在后端内部。调试时能看代码和日志，但产品界面上看不到。

---

## 这次的核心思路

这次改造主要做了两件事：持久化增量摘要、环境信息摘要化。然后再加一层可观测性，让这些动作能在界面上看见。我不想把它做成一个复杂的 Memory 系统。Ward 现在还没走到跨会话长期记忆那一步，也不需要向量库、Recall Memory、Archival Memory 那套完整架构。

它要解决的问题更具体：在已经有上下文注入和运行时摘要的基础上，把历史、环境信息和当前问题组织得更稳定。所以方案也尽量克制。旧历史不要直接丢，压成摘要存下来。环境信息不要全量塞，压成当前轮环境摘要。system prompt 不要每轮变化，保持稳定。最近几轮原文不要动，保护短期连贯性。这几个原则合起来，就是这次升级的主线。

---

## 升级一：持久化增量摘要

以前 Ward 每次新请求进来，只加载最近 20 条消息。这个策略简单，也不会让 prompt 太长。但第 21 条之前的内容没有被总结，直接没了。

如果用户前面说过：

> 后面都按这个口径回答，重点关注风险和下一步动作。

聊到后面，这个偏好可能就被挤出最近 20 条。模型没有看见，自然也就接不上。这次加了一张摘要表。每个 conversation 有一份持久摘要，记录两件事：

- 摘要文本
- 摘要已经覆盖到哪一条 message

加载对话时，结构变成：固定 system prompt、历史摘要、摘要之后的最近原文消息、当前问题。也就是说，旧对话不会静默消失，会变成一份“交接记录”。最近的消息仍然保留原文，因为短期对话最怕被摘要压坏。更早的内容则只保留对继续工作有用的信息。

这里我刻意用了“增量摘要”，避免每次重新总结全部历史。全量摘要的问题是，每次都要把旧历史重新喂给模型，成本高，而且容易出现摘要的摘要。越压越抽象，细节慢慢漂掉。增量摘要更像维护一份持续更新的项目状态：旧摘要加上新离开保护区的消息，生成一份更新后的摘要。每次只处理新增的那一段。摘要覆盖到哪里，也用 message id 记录下来。下一次只看这个边界之后的内容。这套机制不复杂，但它把原来的“最近历史截取”，升级成了“摘要 + 近端原文”的组合。

## 升级二：环境信息摘要化

带工作台的 Agent 和普通聊天工具不太一样。用户问问题时，旁边往往已经有一堆业务对象和中间结果。对 Ward 来说，这些界面上显示的信息本质上就是 Agent 当前可见的外部环境。以前的思路是：环境里有什么，就尽量告诉模型。这样模型可以复用已有信息，少调工具。这次保留这个方向，但换了表达方式。

环境信息不再作为大块原始上下文注入 system prompt。服务端先生成一份环境摘要，再和当前用户问题放在一起，靠后进入 prompt。

举个更通用的例子，一个工作台环境里可能有原始列表、筛选条件、若干工具结果和一份自动生成的报告。模型多数时候真正需要的是：

- 当前用户正在看的对象
- 关键字段和结论
- 数据覆盖范围
- 已经生成过的分析要点
- 哪些信息可能已经过期

原始数据和长报告未必需要全文塞进去。先压成几句摘要，告诉模型“当前环境里已经有过这些信息”，够用就用，不够再调工具。

这有两个好处。第一，模型看到的是更干净的信号。它不用在大段 JSON、表格、日志和长报告里找重点，直接读摘要即可。第二，system prompt 可以稳定下来。之前环境信息拼在 system prompt 里，每轮环境状态一变，system prompt 就变。对 prompt cache 来说，这很不友好。缓存要命中，最重要的是前缀稳定。system prompt、工具定义这些内容应该尽量不动；当前环境摘要、当前问题这类高频变化内容应该往后放。

这次调整以后，Ward 的 prompt 结构更接近：固定规则、历史摘要、最近原文、当前环境摘要、当前用户问题。这个顺序会直接影响 prompt cache。稳定内容尽量稳定，变化内容尽量靠后，缓存才有机会命中。

## 升级三：保护最近几轮原文

上下文压缩很容易犯一个错：为了省 token，把所有历史都拿去摘要。听起来合理，但实际很危险。

用户最近几轮的表达，往往包含大量隐含信息。比如“继续”“按刚才那个口径再看一下”“这次只关注风险”。这些话单独看没意义，必须连着前几轮读。所以 Ward 这次保留了一个最近保护区。还在保护区内的消息不参与摘要，原文留在 prompt 里。只有离开保护区的消息，才会进入增量摘要流程。

这其实是一个很朴素的原则：远端靠摘要，近端靠原文。远端历史负责保留长期状态，近端原文负责维持对话手感。两者不能互相替代。很多 Agent 用起来“断片”，原因经常出在上下文策略上。刚说过的话被压掉，模型下一轮就开始像换了个人。保护区就是为了避免这种断裂。

---

## 升级四：把上下文管理展示出来

这次我特别想把可观测性做出来。因为 Ward 原来已经有上下文管理，但它主要发生在后端。用户看不到，开发时也很难判断它到底有没有按预期工作。

Ward 现在会在聊天气泡顶部显示几类状态：历史加载、环境摘要、摘要检查、增量摘要。每一条状态会带上对应的 token、消息数量或摘要边界。这些文字对普通用户可能可有可无，但对我这个项目来说很重要。Ward 本身就是一个学习型项目。很多功能除了“能用”，也要把 Agent 产品里的机制摊开来看。

以前用户只能看到最终回答。现在至少可以看到：

- 这轮有没有加载历史摘要
- 最近保留了多少条原文
- 环境信息被压到了多大
- 新增历史有没有达到摘要阈值
- 摘要覆盖到了哪条消息

![Ward 上下文管理状态条](/static/2026-06-11/ward-context-events.png)

这有点像之前给 AI 分析做 Runtime Trace。Trace 不会让模型更聪明，但它让系统没那么玄学。上下文管理也是一样。它不能只停留在后台优化，最好能被观察、被解释、被调参。

---

## Prompt Cache 的取舍

这次改动还有一个隐含目标：尽量提高 prompt cache 命中。prompt cache 的核心很简单：前缀越稳定，越容易命中。

但 Agent 对话天然是变化的。用户每轮都在说新话，环境状态也会变，工具结果也可能变。想让所有内容都稳定，不现实。

所以更实际的做法是分层：

- system prompt 固定。
- 工具定义固定。
- 历史摘要低频更新。
- 最近原文按消息顺序追加。
- 当前环境摘要和当前问题放在后面。

高频变化内容不要放在前面。尤其不要每轮改 system prompt。历史摘要不是绝对稳定内容，增量更新时它仍然会变化；真正应该长期稳定的是 system prompt 和工具定义。

![Ward Prompt 结构调整](/static/2026-06-11/ward-context-flow.png)

这听起来像一个底层优化，但它会反过来影响架构设计。你会开始关心：哪些内容是稳定前缀？哪些内容是当前轮变量？哪些内容应该被摘要成冻结状态？哪些内容必须保持原文？prompt cache 当然能省钱，它更大的价值是逼着系统把上下文结构想清楚。

---

## 没有一步到位

这次并没有把上下文管理做到很完整。还有很多可以继续做的东西：

- 工具结果分级压缩
- 更细粒度的摘要段冻结
- 可回溯的原始历史检索
- 摘要质量评估
- 长期用户偏好记忆
- 和 Runtime Trace 打通，统计真实节省了多少 token
- 按真实 token usage 调整摘要阈值

但我不想一开始就上太重。Ward 现在最需要的是把主路径跑顺：长对话不要丢上下文，环境信息不要撑爆 prompt，用户能看到系统怎么处理上下文。先把这三件事做好，比设计一个看起来很完整的 Memory 架构更有意义。

个人项目有个好处：可以很诚实地承认“现在够用”。不需要为了架构完整性引入一堆暂时用不上的东西。

---

## 总结

这次升级以后，Ward 的智能问答把原来已有的上下文管理机制补得更完整了。原来是最近历史 + 环境信息 + 运行时摘要。现在变成了持久摘要 + 近端保护区 + 环境摘要 + 可观测状态。

效果也更明确：

- 长对话里，旧约束不会因为超过最近 20 条就直接消失。
- 环境信息不再把 system prompt 变成动态内容。
- 当前轮 prompt 更清楚地区分稳定前缀和动态变量。
- 用户能看到这轮到底加载了什么、压缩了多少、有没有触发摘要。

上下文管理是个不断迭代的过程，它会随着产品形态继续演进。早期只需要“把环境信息告诉模型”。后来有了工具调用和长任务，就需要运行时摘要。再往后，聊天历史开始变长、环境信息越来越多，就需要持久摘要和环境摘要。如果要做跨会话偏好记忆，又会是另一层设计。

这次做的是把 Ward 原来那套上下文管理，从能用推进到更稳、更省、更可观察。
