# 讲清楚 Harness：一套简单可落地的 LLM Agent 实践

LLM主要应用还是chatbot的时候，大家在讨论 Prompt Engineering，如何设计一个精美的系统提示词，让大模型输出期望的结果。后来大家开始通过多轮会话解决复杂的问题，上下文窗口不够用了，开始强调 Context Engineering，讲究如何组织上下文。最近又开始讨论 Harness Engineering，下面我将结合 OpenAI 和 Anthropic 的文章，聊一下我所理解的 Harness。

**Harness** 这个词，本意是马具——用来控制马匹行动的装置。它的作用不是让马跑得更快，而是让马在正确的方向上、以可控的方式前进。

在 Agent 开发领域，harness 就是让 agent 变得更加**透明、可控，并且能够自己纠错**。它的目标，是把一个原本像"黑盒随机机器"一样的 LLM agent，约束成一个可以稳定完成任务的系统。

换句话说，harness 想解决的问题不是"如何让模型更聪明"，而是：

* 如何让 agent **不那么黑盒**
* 如何让 agent **在犯错时能被及时纠正**
* 如何让 agent **持续朝着任务目标推进**
* 如何让 agent **在合适的时候停止**

最终，agent 不再像一个不可控的随机生成器，而更像一个被约束的 **任务完成器**。

## Harness 背后的核心思路

为了实现这个目标，OpenAI 和 Anthropic 都结合自己的实践，总结出了一些经验。如果仔细看，会发现这些经验和人类在传统软件工程中的很多原则其实高度一致，它只是把人类工程实践中的一些成熟经验，迁移到了 LLM agent 系统里：

* 给执行者提供清晰的环境和工具
* 让系统状态更容易被观察和读取
* 通过外部反馈而不是"自我感觉"来判断结果
* 把复杂任务拆成角色分工明确的流程
* 用日志、状态和规则来约束执行过程

## OpenAI 的建议：让环境和反馈来约束 Agent

OpenAI 的思路更偏向工程系统本身。核心思想可以概括为一句话：

> 与其把精力花在 prompt 打磨上，不如先把 agent 所处的环境设计好。

### 1. 环境优于 Prompt 设计

对于 agent 来说，真正重要的不是一句"更聪明的提示词"，而是它能否随时获取世界的状态，并在环境中执行动作。

一个更适合 agent 工作的环境，通常包括：

* 文件系统视图
* 可执行工具
* 测试用例
* 日志系统
* metrics 监控系统

其实，人类做软件项目时也是一样的。开发者并不是靠一段 prompt 工作，而是依赖编辑器、终端、测试、日志、监控这些基础设施来不断确认自己当前处在什么状态、刚刚做了什么、接下来该做什么。LLM agent 也是如此，如果没有一个良好的环境，再强的模型也很容易变成"凭感觉瞎猜"。

### 2. 输出要对 Agent 友好：尽量结构化、可解析

如果希望 agent 能持续运行，并且不打断自己的执行循环，那么环境给它的反馈就必须足够清晰、可解析。

例如，下面这种结构化输出就比一句模糊的报错更适合 agent 使用：

```json
{
  "status": "failed",
  "error_type": "missing_dependency",
  "file": "auth.py",
  "line": 42
}
```

而不是：

```
build fail error line 42
```

前者的好处在于：

* agent 可以直接 parse
* 更容易形成自动化的反馈闭环
* 更适合后续做重试、分支处理和状态追踪

本质上，不结构化的输出会打断 agent 的 event loop，而结构化输出则更容易让系统维持连续运转。

### 3. 用 Tests 作为判断依据，把"正确性"外包给现实系统

OpenAI 的另一个重要建议是：

> 不要让 LLM 自己判断结果是否合格，而要把"正确性"交给外部系统来验证。

最典型的方式就是使用测试：单元测试、集成测试。这样做的意义在于，agent 不再依赖"自我感觉良好"来宣布任务完成，而是必须面对现实系统给出的反馈。

也就是说，判断标准不再是：

> "我觉得这段代码应该可以。"

而是：

> "测试通过了，所以它暂时是可接受的。"

这一步非常关键，因为它把 agent 的行为从"语言层面的合理"拉回到了"现实系统里的可验证"。

### 4. 强制短反馈循环，尽量让每一步都有反馈

OpenAI 还强调，agent 不应该长时间闷头规划、一次性做很多步，而应该进入一种更短、更紧的反馈循环。

理想状态是：

* agent 做一步动作
* 环境立即返回结果
* agent 根据结果修正下一步动作

也就是说，系统最好形成这样的循环：

```
动作 -> 反馈 -> 修正 -> 再动作
```

而不是：

```
长时间思考 -> 一次性执行很多步 -> 最后才发现错了
```

短反馈循环的价值在于，它能让错误尽早暴露、尽早纠正，避免 agent 在错误方向上越走越远。

## Anthropic 的建议：通过角色分离和状态管理来降低错误

相比 OpenAI 更关注环境和反馈，Anthropic 更强调 agent 内部执行结构的设计。它的建议更偏向"如何组织 agent 的工作方式"。

### 1. 明确拆分角色：Worker 和 Reviewer 不要混在一起

Anthropic 的一个重要经验是：不要让同一个 agent 一边生成结果，一边负责严格审查自己的结果。

更合理的方式是拆成两个角色：

* **Worker Agent**：负责执行任务、产出结果
* **Reviewer Agent**：负责审查结果、发现问题、决定是否需要继续迭代

这里最关键的一点是：

> Worker 和 Reviewer 最好做上下文隔离。

也就是说，Reviewer 不应该看到 Worker 生成过程中的思考轨迹和中间推理产物，而应该尽量只看到：任务目标、当前输出结果、外部反馈信息。

这样做的目的，是避免 Reviewer 继承 Worker 的生成偏见，避免"顺着原来的错误思路继续往下走"，最终"重蹈覆辙"。

从概率分布的角度看，这相当于让 Reviewer 站在一个新的起点上重新判断，而不是在原有思路上做温和修补。

### 2. 明确停止条件，防止过早结束或无限循环

Agent 系统的另一个常见问题是：

* 过早宣布任务完成
* 明明已经够好了还在无休止优化
* 陷入循环，不知道什么时候该停

因此，Anthropic 特别强调要为任务设置**明确的停止条件**。

常见的停止条件包括：

* 测试用例全部通过
* Reviewer Agent 的评分高于某个阈值
* 迭代轮次达到上限
* 某些关键指标达到预期目标

有了明确的停止条件，agent 的事件循环才真正可控。

### 3. 外化记忆：不要把所有信息都塞进上下文窗口

Anthropic 还非常强调 **external memory**，也就是把关键状态从上下文窗口中拿出来，持久化到外部存储中。

例如，可以把这些信息写入文件或持久化状态中：

* 任务状态
* 处理日志
* 推理历史
* 当前决策
* 已完成步骤
* 待办事项

这样做的好处有两个：

第一，上下文窗口是有限的。如果把所有内容都塞进上下文里，迟早会遇到窗口不够用的问题。

第二，上下文过载会分散模型注意力。即使窗口还没满，信息过多也会让 agent 更难聚焦在当前最重要的问题上。

因此，更合理的做法是：把长期状态放到外部，在必要时按需读取，把上下文窗口更多留给当前任务所需的工作记忆。

## 总结：Harness 只是把工程经验迁移到了 Agent 系统里

如果把 OpenAI 和 Anthropic 的建议放在一起看，会发现 harness 的核心并不复杂：

* 给 agent 一个足够好的执行环境
* 让系统反馈尽量结构化、可解析
* 用外部测试和真实反馈来判断结果
* 通过角色拆分降低自我偏见
* 用明确的停止条件防止失控
* 通过外化记忆管理长期状态

以上这些点并不是 harness 的全部，业界可能在实践中不断发掘更多让 Agent 可控的方案，但无论是什么方案，harness 做的事情并不是"消灭随机性"，而是**约束随机性**，它把 LLM 这个随机系统限制在一个更可控、更透明、更容易纠错的框架里。
